# 19. 페이지별 기능 정의서

16-기능정의서.md의 31개 고유 기능을 페이지 단위로 재편성. 배치/수동 생성 기능은 해당 페이지에 포함.
08-사용자-행동별-기능정의.md의 사용자 플로우·입력 규칙·노출 데이터를 본 문서에 통합.

**범위**: 인증 제외, 팀장/팀원 역할 구분 없음 (MVP)
**고유 기능 31개** | 페이지별 중복 포함 시 37개 | API 약 25개 | 페이지 11개 (팀 공유 탭 3개 포함)

> **[프로토타입 안내]** 현재 프론트엔드 프로토타입은 백엔드 API 없이 **localStorage**로 모든 데이터를 저장/조회합니다. 최초 접속 시 이름 입력으로 사용자 등록 후 사용합니다. 각 기능별 프로토타입 구현 방식은 `[프로토타입]` 태그로 표기합니다.

## 사용자 플로우 요약

```
1. 최초 접속 → 팀 생성(팀장) 또는 팀 등록(팀원) → 메인으로 이동
2. 메인(데일리 슈크럼 작성) → 저장 → 데일리 보고 탭에서 확인
3. 배치/수동으로 주간·월간 보고 자동 생성 → AI 요약(월간·성과)
4. 팀 업무 공유에서 팀원별 보고 조회 (민감정보 제외)
```

**좌측 메뉴 구조**:
- **[데일리 슈크럼 작성]** (/ , 메인) — 버튼형 CTA. 서비스 핵심 기능으로 눈에 띄게 표시
- **개인** 섹션
  - 일간 보고 (/reports/daily)
  - 주간 보고 (/reports/weekly)
  - 월간 보고 (/reports/monthly)
  - 성과 보고 (/reports/review)
- **팀 공유** 섹션 (조회 전용 — 팀원들의 보고를 공유/열람하는 페이지)
  - 일간 보고 (/team/share?tab=daily)
  - 주간 보고 (/team/share?tab=weekly)
  - 월간 보고 (/team/share?tab=monthly)
- **[팀설정 아이콘]** 헤더 우측 톱니바퀴(⚙) 아이콘 + "팀 설정" 텍스트 클릭 시 드롭다운 표시
  - 팀이 없는 경우: "팀을 먼저 생성하거나 팀원들에게 멤버등록을 요청하세요" + 확인 버튼 (닫기만)
  - 팀이 있는 경우: 팀원 등록하기, 팀 정보 세팅 메뉴 노출

**팀 미지정 시 접근 제한**:
- 팀이 없는 상태에서 본문 영역에 "팀을 먼저 생성하거나 팀원들에게 멤버등록을 요청하세요." + 팀 생성하기 버튼 표시
- 좌측 메뉴 클릭 시에도 동일 메시지 alert 표시 (데일리 슈크럼 작성, 개인 보고, 팀 공유 메뉴 모두)
- 팀설정 페이지(팀 생성, 팀 등록, 팀원 등록, 태그 설정)만 접근 가능

**관리자 여부 (member.managerYn)**:
- `Y` (관리자/팀장): 팀 설정 수정 가능, 팀원 등록/삭제 가능
- `N` (일반 팀원): 팀 설정은 읽기 전용 조회만 가능, 단 팀원 등록는 가능 (삭제 불가)
- 팀 생성 시 생성자의 managerYn은 자동으로 'Y' 설정
- DDL: member 테이블에 `manager_yn CHAR(1) NOT NULL DEFAULT 'N'` 컬럼 추가

## 1. 팀 생성 페이지

**경로**: /team/create | **좌측 메뉴**: 팀 생성하기

| # | 기능 | FE | BE | 상세 |
|---|------|:--:|:--:|------|
| T-1 | 팀 생성 | O | O | 팀명 입력 → 저장. **1인당 1팀만 생성 가능. 생성 시 팀장이 됨.** 이미 생성한 팀이 있으면 "이미 생성된 팀이 있습니다" + 팀명 표시 + **"팀원 등록하러 가기" 버튼**(/team/members로 이동). 필요 시 팀 삭제 버튼으로 팀 삭제 가능. 팀원 등록은 팀장만 가능. |

**노출 필드**: team.teamName
**저장 필드**: team(teamName)
**API**: POST /teams, DELETE /teams/{teamId}
**[프로토타입]** localStorage 저장. 실서비스에서 REST API로 전환.

**팀 생성 시 동작**:
- 팀 생성자 = 팀장. 생성 즉시 본인의 member.teamId 자동 배정 + currentTeamId 설정

**팀 삭제 시 동작**:
- 해당 팀에 속한 멤버의 teamId → NULL 처리
- 삭제한 사용자의 currentTeamId도 해제
- 삭제 전 확인 다이얼로그 표시

## 3. 팀 등록 페이지

**경로**: /team/assign | **좌측 메뉴**: 팀 등록하기 (팀 미등록 시에만 노출)

| # | 기능 | FE | BE | 상세 |
|---|------|:--:|:--:|------|
| T-5 | 팀 목록 검색 | O | O | 팀 이름 자동 검색 (입력 즉시 필터링) |
| T-6 | 팀 등록 (본인 배정) | O | O | 선택 시 member.teamId 갱신. 이미 팀이 있으면 "현재 스프린톤에서는 다중 팀 미지원. 추후 개선 예정" 에러 |

**Member 생성 시점**:
- **본인 팀 등록(T-6)**: 로그인(인증) 시 Dooray API로 사용자 정보 조회하여 Member가 없으면 자동 생성. 팀 등록은 이미 존재하는 Member의 teamId만 갱신.
- **팀원 등록(T-7)**: 이름 검색 후 선택한 대상이 Member 테이블에 없으면 **Dooray API로 해당 사용자 정보를 조회하여 Member를 새로 생성**한 뒤 teamId를 등록. 이미 Member가 존재하면 teamId만 갱신.

**노출 필드**: team.teamId, team.teamName
**저장 필드**: member(teamId)
**API**: GET /teams?q=, PUT /members/me/team
**[프로토타입]** localStorage에서 팀 목록 검색 및 member.teamId 갱신. 실서비스에서 REST API로 전환.
**팀 미지정 시**: 메인에 "팀을 지정해주세요" 빈 상태 표시
**이미 지정된 팀이 있을 때**: 현재 지정된 팀명만 표시

## 5. 데일리 슈크럼 작성 페이지

**경로**: / (메인) | **탭**: 데일리 슈크럼 작성 (기본 활성)

**상단 탭 구조** (5개 탭, 모든 보고 페이지에서 공유):

| 탭 | 경로 | 출처 |
|----|------|------|
| 데일리 슈크럼 작성 | / | 작성 화면. 저장 시 report.type=daily |
| 데일리 보고 | /reports/daily | report.type=daily |
| 주간 보고 | /reports/weekly | report.type=weekly |
| 월간 보고 | /reports/monthly | report.type=monthly |
| 성과 보고 | /reports/review | report.type=review |

| # | 기능 | FE | BE | 상세 |
|---|------|:--:|:--:|------|
| D-1 | 두레이 업무 목록 자동 조회 | O | O | WAS가 Dooray API 대리 호출 (FE → BE → Dooray API). 당일 멘션/댓글/담당 업무. **[프로토타입] 수동 입력만 지원. 실서비스에서 Dooray API 연동** |
| D-2 | 데일리 슈크럼 저장 | O | O | report(type=daily) + reportDetail 저장. 당일 기존 데이터 있으면 덮어쓰기(수정) |
| D-3 | 행 직접 추가 (+) | O | - | 시스템 목록 외 업무 수동 추가 |
| D-4 | 행 제거 (X) | O | O | 저장 전 UI제거, 저장 후 삭제 API |
| D-5 | 주간 보고 태그 선택 (필수) | O | - | 미선택 시 저장 불가. 월간 태그는 parentTagId로 시스템 자동 매핑 (화면에 읽기전용 표시) |
| D-6 | 데일리 슈크럼 수정 | O | O | 저장 후에도 행 수정·삭제 가능 |
| D-8 | 동료 협업 기록 저장 | O | O | "동료 협업 기록" 버튼 → 모달. 팀 동료 이름 선택 → 내용 작성 → 저장. 여러 동료 추가 가능 |
| D-9 | 동료 협업 기록 수정/삭제 | O | O | 모달 내에서 기존 기록 수정/삭제 |

**노출 필드**:
- 업무 행: reportDetail.taskTitle(업무 제목), reportDetail.done(한 일), reportDetail.workHours(투입시간), reportDetail.performance(성과), reportDetail.taskLink(업무 링크)
- 태그: reportDetail.tagId → tag.tagName(주간), tag.parentTagId → 월간 태그(자동 매핑, 읽기전용)
- 동료: member.memberName(팀 소속 동료만), peerReport.content

**저장 필드**: report(memberId, staDate, endDate, type=daily), reportDetail(reportId, tagId, taskId, taskTitle, taskLink, done, workHours, performance), peerReport(reportId, peerMemberId, content)

**API**: GET /dooray/tasks, POST /reports/daily, PUT /reports/daily/{reportId}, DELETE /reports/daily/{reportId}/details/{detailId}, POST/PUT/DELETE /reports/daily/{reportId}/peer-reports
**[프로토타입]** 모든 저장/조회를 localStorage로 처리. 업무 제목은 사용자가 직접 입력 (작성 화면에서도 수정 가능). 실서비스에서 REST API + Dooray API로 전환.

**Dooray API 호출 방식**: FE가 JWT에 포함된 암호화된 dooray_api_token을 요청 헤더로 전달 → BE가 복호화 후 Dooray API 대리 호출 → 결과 반환. **BE DB에 Dooray 토큰을 직접 저장하지 않음.**

**입력 규칙**:
- **업무 제목(taskTitle)**: 데일리 슈크럼 **작성** 화면에서는 **수정 불가**. 링크 입력 시 백엔드가 Dooray API로 제목 조회 후 자동 입력. 저장 후 데일리 보고·수정 화면에서는 **수정 가능**
- **한 일(done)**: **필수**, 공백/빈문자 불가, 최소 1글자
- **투입시간(workHours)**: 소수점 불가, 1시간 단위, **1~24시간(정수)**
- **업무 링크(taskLink)**: 사용자 입력·수정 가능 (URL). 있으면 "새 탭에서 열기" 링크 표시
- **주간 태그**: **필수**. 미선택 시 저장 불가
- **어필할 성과(performance)**: 선택. **클릭 시 팝업(모달)에서 입력** (인라인 아님)
- **동료 협업 기록**: 데일리 보고(type=daily)에만 저장 가능. 주간/월간/성과 보고에는 불가.

|---|------|:--:|:--:|------|

## 8. 월간 보고 페이지 (GRM)

**경로**: /reports/monthly | **탭**: 월간 보고

| B-2 | 월간 보고 자동 생성 | - | O | 매월 25일 5AM, 1~25일 구간 |
| R-6a | 월간 보고 수동 생성 | O | O | 화면 상단 "월간 보고 생성" 버튼 |

- 태그 기준 그룹핑 → 합산 → AI 요약 생성(태그별 200자) → report + reportDetail + reportTag(aiSummaryContent) 저장
- **26~말일 데이터**: 배치에 포함되지 않음. 수동 재생성 시 해당 시점까지의 데이터로 다시 생성 가능.
- **1 멤버 1 트랜잭션**: 특정 멤버 실패 시 다른 멤버에 영향 없음

**AI 요약** (A-1):
- 모델: Claude API (Sonnet)
- 입력: done, workHours, performance, taskTitle 태그별 합친 텍스트
- 출력: 200자 내외
- **초과 정책**: 시스템 프롬프트에서 글자수 제한 지시. 초과 시 잘라내기 적용.
- 사용자 업무 데이터만 기반으로 요약. 원본 업무 링크 병기하여 검증 가능성 확보.

## 공통 규칙

**모든 보고 수정 허용**: 원래 수정 불가로 계획했으나, 상위 레벨 데이터 재생성 시 반영을 위해 **모든 보고 타입에서 수정 허용**.

**배치 트랜잭션 전략**: 1 멤버 1 트랜잭션. 특정 멤버 처리 실패 시 다른 멤버에 영향 없음.

**Dooray API 호출 경로**: FE → BE(WAS) → Dooray API. FE가 직접 Dooray API를 호출하지 않음. JWT claims에 암호화된 dooray_api_token 포함, BE가 복호화 후 대리 호출.

**AI 요약 공통**: Claude API(Sonnet) 사용. 사용자 업무 데이터만 기반 요약 (할루시네이션 방지). 요약 결과에 원본 업무 링크 병기 (검증 가능성 확보). 초과 시 잘라내기 적용.
