# 19. 페이지별 기능 정의서

16-기능정의서.md의 31개 고유 기능을 페이지 단위로 재편성. 배치/수동 생성 기능은 해당 페이지에 포함.
08-사용자-행동별-기능정의.md의 사용자 플로우·입력 규칙·노출 데이터를 본 문서에 통합.

**범위**: 인증 제외, 팀장/팀원 역할 구분 없음 (MVP)
**고유 기능 31개** | 페이지별 중복 포함 시 37개 | API 약 25개 | 페이지 11개 (팀 공유 탭 3개 포함)

> **[프로토타입 안내]** 현재 프론트엔드 프로토타입은 백엔드 API 없이 **localStorage**로 모든 데이터를 저장/조회합니다. 최초 접속 시 이름 입력으로 사용자 등록 후 사용합니다. 각 기능별 프로토타입 구현 방식은 `[프로토타입]` 태그로 표기합니다.

## 사용자 플로우 요약

```
1. 최초 접속 → 팀 생성(팀장) 또는 팀 등록(팀원) → 메인으로 이동
2. 메인(데일리 슈크럼 작성) → 저장 → 데일리 보고 탭에서 확인
3. 배치/수동으로 주간·월간 보고 자동 생성 → AI 요약(월간·성과)
4. 팀 업무 공유에서 팀원별 보고 조회 (민감정보 제외)
```

**좌측 메뉴 구조** (현재 위치에 해당하는 메뉴 항목은 파란색 활성 상태로 표시):
- **[데일리 슈크럼 작성]** (/ , 메인) — 버튼형 CTA. 서비스 핵심 기능으로 눈에 띄게 표시
- **개인** 섹션
  - 일간 보고 (/reports/daily)
  - 주간 보고 (/reports/weekly)
  - 월간 보고 (/reports/monthly)
  - 성과 보고 (/reports/review)
- **팀 공유** 섹션 (조회 전용 — 팀원들의 보고를 공유/열람하는 페이지)
  - 일간 보고 (/team/share?tab=daily)
  - 주간 보고 (/team/share?tab=weekly)
  - 월간 보고 (/team/share?tab=monthly)
- **[팀설정 아이콘]** 헤더 우측 톱니바퀴(⚙) 아이콘 + "팀 설정" 텍스트 클릭 시 드롭다운 표시
  - 팀이 없는 경우: "팀을 먼저 생성하거나 팀원들에게 멤버등록을 요청하세요" + 확인 버튼 (닫기만)
  - 팀이 있는 경우: 팀원 등록하기, 팀 정보 세팅 메뉴 노출

**팀 미지정 시 접근 제한**:
- 팀이 없는 상태에서 본문 영역에 "팀을 먼저 생성하거나 팀원들에게 멤버등록을 요청하세요." + 팀 생성하기 버튼 표시
- 좌측 메뉴 클릭 시에도 동일 메시지 alert 표시 (데일리 슈크럼 작성, 개인 보고, 팀 공유 메뉴 모두)
- 팀설정 페이지(팀 생성, 팀 등록, 팀원 등록, 태그 설정)만 접근 가능

**관리자 여부 (member.managerYn)**:
- `Y` (관리자/팀장): 팀 설정 수정 가능, 팀원 등록/삭제 가능
- `N` (일반 팀원): 팀 설정은 읽기 전용 조회만 가능, 단 팀원 등록은 가능 (삭제 불가)
- 팀 생성 시 생성자의 managerYn은 자동으로 'Y' 설정
- DDL: member 테이블에 `manager_yn CHAR(1) NOT NULL DEFAULT 'N'` 컬럼 추가

**경로**: /team/create | **좌측 메뉴**: 팀 생성하기

| # | 기능 | FE | BE | 상세 |
|---|------|:--:|:--:|------|
| T-1 | 팀 생성 | O | O | 팀명 입력 → 저장. **1인당 1팀만 생성 가능. 생성 시 팀장이 됨.** 이미 생성한 팀이 있으면 "이미 생성된 팀이 있습니다" + 팀명 표시 + **"팀원 등록하러 가기" 버튼**(/team/members로 이동). 필요 시 팀 삭제 버튼으로 팀 삭제 가능. 팀원 등록은 팀장만 가능. |

**노출 필드**: team.teamName
**저장 필드**: team(teamName)
**API**: POST /teams, DELETE /teams/{teamId}
**[프로토타입]** localStorage 저장. 실서비스에서 REST API로 전환.

**팀 생성 시 동작**:
- 팀 생성자 = 팀장. 생성 즉시 본인의 member.teamId 자동 배정 + currentTeamId 설정
- 팀 생성자의 member.managerYn = 'Y' (관리자) 자동 설정
- 생성 후 "팀원 등록하러 가기" 버튼으로 /team/members 이동 유도

**팀 삭제 시 동작**:
- 해당 팀에 속한 멤버의 teamId → NULL 처리
- 삭제한 사용자의 currentTeamId도 해제
- 삭제 전 확인 다이얼로그 표시

## 2. 팀 관리 페이지 (설정)

**경로**: /team/manage | **좌측 메뉴**: 팀 정보 세팅

**진입 조건**: 팀이 등록되어 있어야 접근 가능. 미등록 시 "속한 팀을 먼저 등록해주세요" + 팀 등록하기 링크 표시.

| # | 기능 | FE | BE | 상세 |
|---|------|:--:|:--:|------|
| T-2 | 주간 보고 시작/종료 요일 설정 | O | O | 드롭다운 (0=일~6=토). 시작~종료가 정확히 일주일(7일)이 되어야 저장 가능. |

**주간 보고 요일 밸리데이션 (FE + BE 공통)**:
- 시작 요일과 종료 요일 모두 필수 선택
- 시작~종료가 정확히 일주일이 되어야 함 (예: 월~일, 수~화, 금~목)
- 종료 요일 = (시작 요일 + 6) % 7 이어야 유효
- 조건 불충족 시 FE: 에러 메시지 표시, BE: **400 Bad Request** 오류 반환
| T-3 | 주간 보고 태그 CRUD | O | O | type=weekly. 추가/수정/삭제. 태그명 입력 + 부모(월간) 태그 선택 |
| T-4 | 월간(ESM) 태그 CRUD | O | O | type=monthly. 주간 태그의 부모. 추가/수정/삭제 |

**태그 계층 구조**:
- 월간 태그(parentTagId=NULL): 상위 그룹 (예: ESM 프로젝트명)
- 주간 태그(parentTagId=월간태그ID): 하위 분류 (예: 세부 업무 분류)
- 주간 태그는 반드시 월간 태그 하위에 속해야 함 (미분류 허용 안 함)

**태그 저장 방식**:
- 저장 버튼 클릭 시 해당 팀의 **기존 월간/주간 태그를 전부 삭제 후 다시 일괄 저장** (Replace All 방식)
- API: `PUT /teams/{teamId}/tags` — Request Body에 월간+주간 태그 트리 전체를 전달
- 백엔드에서도 월간 태그 내에 주간 태그가 1개 이상 없으면 **400 Bad Request** 오류 반환

**태그 저장 밸리데이션 (FE + BE 공통)**:
- 월간 태그명이 비어있으면 저장 불가
- 월간 태그가 있는데 하위 주간 태그가 없으면 저장 불가
- 주간 태그명이 비어있으면 저장 불가
- 월간 태그 없이 주간 태그만 존재할 수 없음

**노출 필드**: team.teamName, team.weekStartDay, team.weekEndDay, tag.tagName, tag.type, tag.parentTagId
**저장 필드**: team(weekStartDay, weekEndDay), tag(teamId, parentTagId, tagName, type)
**API**: PUT /teams/{teamId}/schedule, PUT /teams/{teamId}/tags (일괄 저장)
**[프로토타입]** localStorage 저장. 실서비스에서 REST API로 전환.

## 3. 팀 등록 페이지

**경로**: /team/assign | **좌측 메뉴**: 팀 등록하기 (팀 미등록 시에만 노출)

| # | 기능 | FE | BE | 상세 |
|---|------|:--:|:--:|------|
| T-5 | 팀 목록 검색 | O | O | 팀 이름 자동 검색 (입력 즉시 필터링) |
| T-6 | 팀 등록 (본인 배정) | O | O | 선택 시 member.teamId 갱신. 이미 팀이 있으면 "현재 스프린톤에서는 다중 팀 미지원. 추후 개선 예정" 에러 |

**Member 생성 시점**:
- **본인 팀 등록(T-6)**: 로그인(인증) 시 Dooray API로 사용자 정보 조회하여 Member가 없으면 자동 생성. 팀 등록은 이미 존재하는 Member의 teamId만 갱신.
- **팀원 등록(T-7)**: 이름 검색 후 선택한 대상이 Member 테이블에 없으면 **Dooray API로 해당 사용자 정보를 조회하여 Member를 새로 생성**한 뒤 teamId를 등록. 이미 Member가 존재하면 teamId만 갱신.

**노출 필드**: team.teamId, team.teamName
**저장 필드**: member(teamId)
**API**: GET /teams?q=, PUT /members/me/team
**[프로토타입]** localStorage에서 팀 목록 검색 및 member.teamId 갱신. 실서비스에서 REST API로 전환.
**팀 미등록 시**: 메인에 "팀을 먼저 생성하거나 팀원들에게 멤버등록을 요청하세요." 빈 상태 표시
**이미 등록된 팀이 있을 때**: 현재 등록된 팀명만 표시

## 5. 데일리 슈크럼 작성 페이지

**경로**: / (메인) | **탭**: 데일리 슈크럼 작성 (기본 활성)

**상단 탭 구조** (5개 탭, 데일리 슈크럼 작성 페이지 포함 모든 보고 페이지에서 공유):

| 탭 | 경로 | 출처 |
|----|------|------|
| 데일리 슈크럼 작성 | / | 작성 화면. 저장 시 report.type=daily |
| 일간 보고 | /reports/daily | report.type=daily |
| 주간 보고 | /reports/weekly | report.type=weekly |
| 월간 보고 | /reports/monthly | report.type=monthly |
| 성과 보고 | /reports/review | report.type=review |

**진입 조건 (팀 설정 완료 검증)**:
- 데일리 슈크럼 작성 페이지 진입 시 아래 3가지 항목을 모두 검증
  1. 주간 보고 시작/종료 요일이 설정되어 있는지 (`team.weekStartDay`, `team.weekEndDay`)
  2. 월간 태그(ESM명)가 1개 이상 존재하는지
  3. 주간 태그가 1개 이상 존재하는지
- **하나라도 미설정 시**: 팝업(모달) 표시 — "팀 설정이 필요합니다" + 미설정 항목 목록 + **"팀 정보 세팅하러 가기" 버튼** (`/team/manage`로 이동)
- 팝업이 떠 있는 동안 작성 폼은 비활성 상태 (배경에 표시)
- BE에서도 데일리 슈크럼 저장 API 호출 시 동일 조건 검증 → 미충족 시 **400 Bad Request** 반환

| # | 기능 | FE | BE | 상세 |
|---|------|:--:|:--:|------|
| D-1 | 두레이 업무 목록 자동 조회 | O | O | WAS가 Dooray API 대리 호출 (FE → BE → Dooray API). 당일 멘션/댓글/담당 업무. **[프로토타입] 수동 입력만 지원. 실서비스에서 Dooray API 연동** |
| D-1a | 업무 링크 → 제목 자동 조회 | O | O | 업무 링크(URL) 입력 시 BE가 Dooray API를 호출하여 업무 제목(taskTitle)을 자동으로 채워줌. API 응답이 없으면 빈 값. 사용자가 직접 수정 가능 (항상 활성화 상태). **[프로토타입] 자동 조회 없이 사용자가 직접 업무 제목 입력. 업무 제목 필드 하단에 가이드 텍스트 표시.** |
| D-2 | 데일리 슈크럼 저장 | O | O | report(type=daily) + reportDetail 저장. 당일 기존 데이터 있으면 덮어쓰기(수정) |
| D-3 | 행 직접 추가 (+) | O | - | 시스템 목록 외 업무 수동 추가 |
| D-4 | 행 제거 (X) | O | O | 저장 전 UI제거, 저장 후 삭제 API |
| D-5 | 주간 보고 태그 선택 (필수) | O | - | 미선택 시 저장 불가. 월간 태그는 parentTagId로 시스템 자동 매핑 (화면에 읽기전용 표시) |
| D-6 | 데일리 슈크럼 수정 | O | O | 저장 후에도 행 수정·삭제 가능 |
| D-8 | 동료 협업 기록 저장 | O | O | "동료 협업 기록" 버튼 → 모달. 팀 동료 이름 선택 → 내용 작성 → 저장. 여러 동료 추가 가능. **버튼은 작성/읽기 모드 모두 노출** (데일리 저장과 독립적인 별도 CRUD). **같은 날 같은 동료에 대해 중복 기록 불가** (이미 기록된 동료는 선택지에서 제외) |
| D-9 | 동료 협업 기록 수정/삭제 | O | O | 모달 내에서 기존 기록 인라인 수정(수정 버튼 → textarea 편집 → 저장/취소) 및 삭제 |
| D-10 | 내일 할 일 입력 | O | O | report.tomorrowPlan 필드. "내일 할 일" 버튼(초록색) 클릭 → 모달에서 입력 (선택). **데일리 슈크럼 미저장 시 "먼저 데일리 슈크럼을 저장해주세요" alert 표시**. 팀 공유 페이지에서 팀원별로 노출됨 |

**노출 필드**:
- 업무 행: reportDetail.taskTitle(업무 제목), reportDetail.done(한 일), reportDetail.workHours(투입시간), reportDetail.performance(성과), reportDetail.taskLink(업무 링크)
- 태그: reportDetail.tagId → tag.tagName(주간), tag.parentTagId → 월간 태그(자동 매핑, 읽기전용)
- 동료: member.memberName(팀 소속 동료만), peerReport.content

**저장 필드**: report(memberId, staDate, endDate, type=daily, tomorrowPlan), reportDetail(reportId, tagId, taskId, taskTitle, taskLink, done, workHours, performance), peerReport(reportId, peerMemberId, content)

**API**: GET /dooray/tasks, POST /reports/daily, PUT /reports/daily/{reportId}, DELETE /reports/daily/{reportId}/details/{detailId}, POST/PUT/DELETE /reports/daily/{reportId}/peer-reports
**[프로토타입]** 모든 저장/조회를 localStorage로 처리. 업무 제목은 사용자가 직접 입력 (작성 화면에서도 수정 가능). 실서비스에서 REST API + Dooray API로 전환.

**Dooray API 호출 방식**: FE가 JWT에 포함된 암호화된 dooray_api_token을 요청 헤더로 전달 → BE가 복호화 후 Dooray API 대리 호출 → 결과 반환. **BE DB에 Dooray 토큰을 직접 저장하지 않음.**

**입력 규칙**:
- **업무 제목(taskTitle)**: 데일리 슈크럼 **작성** 화면에서는 **수정 불가**. 링크 입력 시 백엔드가 Dooray API로 제목 조회 후 자동 입력. 저장 후 데일리 보고·수정 화면에서는 **수정 가능**
- **한 일(done)**: **필수**, 공백/빈문자 불가, 최소 1글자
- **투입시간(workHours)**: 소수점 불가, 1시간 단위, **1~24시간(정수)**
- **업무 링크(taskLink)**: `https://nhnent.dooray.com/project/tasks/{taskId}` 형식만 허용 (taskId는 정수). 유효하지 않은 URL 입력 시 alert 후 초기화. 입력 필드 하단에 "두레이 업무의 링크 복사 버튼을 통해 복사한 것을 붙여넣기 해주세요" 가이드 표시. 유효한 링크가 있으면 "새 탭에서 열기" 링크 표시
- **주간 태그**: **필수**. 미선택 시 저장 불가
- **어필할 성과(performance)**: 선택. **클릭 시 팝업(모달)에서 입력/수정** (인라인 아님). 데일리 슈크럼 작성 페이지·일간 보고 페이지 모두에서 **수정 가능** (readOnly 제한 없음)
- **동료 협업 기록**: 데일리 보고(type=daily)에만 저장 가능. 주간/월간/성과 보고에는 불가.
  - **독립 CRUD**: 동료 협업 기록은 데일리 슈크럼 저장과 **별도의 독립적인 CRUD**로 동작. 단, **데일리 슈크럼 미저장 시 "먼저 데일리 슈크럼을 저장해주세요" alert 표시** 후 모달 미오픈
  - **API**: GET /reports/daily/peer-reports?date={yyyy-MM-dd} (해당 일자를 파라미터로 전달하여 조회), POST /reports/daily/peer-reports (저장), PUT /reports/daily/peer-reports/{peerReportId} (수정), DELETE /reports/daily/peer-reports/{peerReportId} (삭제)
  - **버튼 노출**: 작성 모드·읽기 모드 모두 "동료 협업 기록" 버튼(보라색) 노출 (우측 상단). "내일 할 일" 버튼(초록색)도 동일 위치에 노출

**생성 로직** (R-7):
- **소스**: **월간보고**의 report_detail **전체 업무** 포함 (performance 유무와 무관). 단, **AI 요약 context 생성 시에만** performance가 있는 업무만 필터링하여 사용
- **날짜 범위**: 사용자가 시작일/종료일 지정 (기본 1/1~12/31). 기존 성과보고와 기간 중복 시 409 오류
- 동일 업무 합치기(task_id 기준): done/performance/aiSummary `\n\n` 연결, workHours 합산 → 최신일자 기준 task_title/task_link/tag_id 사용 → 태그별 정렬 → report + reportDetail 생성
- **업무별 AI 요약** (동일 업무(중복)가 있었던 업무만 대상): 합쳐진 aiSummary를 context로 Claude API 개조식 요약 → report_detail.ai_summary 저장. 중복 없는 업무는 월간보고의 기존 aiSummary 그대로 사용
- **태그별 공수 합산** (성과보고만): **월간 보고의 전체 업무**(performance 유무와 무관) work_hours를 월간 태그 기준으로 합산 → report_detail_tag.work_hours 저장
- **태그별 AI 요약** (필수, performance가 있는 업무만 대상): 태그당 1회 Claude API 호출 → report_detail_tag.ai_summary 저장 (500자 내외)

**AI 요약** (A-2) — 태그별:
- 모델: Claude API (Sonnet)
- 프롬프트 파일 별도 참조
- **context 구성**: 월간 태그 내의 모든 월간 AI 요약(reportDetail.aiSummary) 및 업무 성과(reportDetail.performance)를 합하여 context로 사용
- 입력 context JSON (태그당 1회 호출, context 희석 방지, **performance가 있는 업무만** 포함):
  ```json
  {
    "tagId": 123, "tagName": "프로젝트A",
    "tasks": [
      { "aiSummary": "업무별 월간 AI 요약값", "performance": "사용자가 해당 업무에 대해 어필하고 싶어하는 내용으로 등록한 값" }
    ]
  }
  ```
- 반환 JSON: `{ "tagId": 123, "tagName": "프로젝트A", "aiSummary": "요약 결과..." }`
- 출력: 500자 내외
- **초과 정책**: 시스템 프롬프트에서 글자수 제한 지시. 초과 시 잘라내기 적용.

## 10. 팀 업무 공유 — 데일리 보고

**경로**: /team/share (탭: 데일리 보고, 기본 활성) | **좌측 메뉴**: 팀 업무 공유

| # | 기능 | FE | BE | 상세 |
|---|------|:--:|:--:|------|
| S-1 | 팀원 데일리 보고 목록 조회 | O | O | 민감정보 제외 |
| S-1a | 날짜별 그룹 표시 | O | — | 같은 날짜의 보고를 하나의 카드로 묶어 표시 (최신순 정렬) |
| S-1b | 마이페이지 일간 보고와 동일 컬럼 구조 | O | — | 컬럼: 팀원, 업무명(하이퍼링크), 업무 내용(한 일/done). 작업 버튼 제외 |
| S-1c | 업무명 하이퍼링크 | O | — | taskLink가 유효한 dooray URL이면 새 탭에서 열리는 하이퍼링크로 표시 |
| S-1d | 내일 할 일 표시 | O | O | report.tomorrowPlan 필드. 팀원별 내일 할 일을 마지막 컬럼에 표시 |

**컬럼 구성**: 팀원 | 총 투입시간 | 업무명 (하이퍼링크) | 투입률 (시간+%) | 업무 내용 (한 일/done) | 내일 할 일
- 팀원 컬럼: 동일 팀원의 여러 업무는 rowSpan으로 병합
- 업무명 컬럼: taskLink가 dooray URL이면 `<a>` 하이퍼링크, 아니면 일반 텍스트
- 업무 내용 컬럼: reportDetail.done 필드 (whitespace-pre-line)
- 작업 버튼(상세/수정/삭제/협업)은 표시하지 않음 (조회 전용)

**노출 필드**: member.memberName(회원명), report.staDate/endDate, reportDetail.taskTitle(업무 제목), reportDetail.taskLink(업무 링크), reportDetail.done(한 일)
**노출 제외 (민감)**: reportDetail.performance(성과), peerReport.content(동료 기록), reportDetail.workHours(투입시간)
**API**: GET /teams/{teamId}/reports/daily?date=
**[프로토타입]** localStorage에서 팀원별 데이터 필터링. 날짜별 그룹은 FE에서 Map으로 처리. 실서비스에서 REST API로 전환.

## 공통 규칙

**모든 보고 수정 허용**: 원래 수정 불가로 계획했으나, 상위 레벨 데이터 재생성 시 반영을 위해 **모든 보고 타입에서 수정 허용**.

**배치 트랜잭션 전략**: 1 멤버 1 트랜잭션. 특정 멤버 처리 실패 시 다른 멤버에 영향 없음.

**Dooray API 호출 경로**: FE → BE(WAS) → Dooray API. FE가 직접 Dooray API를 호출하지 않음. JWT claims에 암호화된 dooray_api_token 포함, BE가 복호화 후 대리 호출.

**AI 요약 공통**: Claude API(Sonnet) 사용. 사용자 업무 데이터만 기반 요약 (할루시네이션 방지). 요약 결과에 원본 업무 링크 병기 (검증 가능성 확보). 초과 시 잘라내기 적용.
