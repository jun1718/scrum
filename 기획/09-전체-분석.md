# 9. 전체 분석

## 잘 잡힌 부분

- 문제 정의(시간·인지·통일성), "데일리 하나만 쓰면 다 해준다" 해결 방향, UI·테이블·보고 계층 일치.
- 배치 시점·Member/인증·report.type 정리 반영.

## 반드시 정리한 부분 (반영 완료)

- **배치 시점**: 월간 보고 매월 25일 새벽 5시, 당월 구간. 주간 보고 매일 새벽 3시, 전주.
- **Member vs 인증**: Dooray API + Member 조회/생성, JWT claim(memberId, dooray_api_token 암호화). memberLoginId/memberPassword 제거.
- **report.type**: daily 추가. 데일리 보고 = type=daily report 1건 + reportDetail.

## 보강 반영 사항

- API 후보 목록(11), 두레이 업무 조회(06), AI 요약 규격(07), 에러·유효성 메시지(12), ERD 논리 검토·컬럼 한글 매핑(13), 1일 개발 준비 체크리스트(15).

## 리스크·추가 보강

- 두레이 API 의존: 업무 목록·멤버 조회 스펙·권한 확인 필요.
- 1일/2일 스코프 시 MVP 범위(데일리+주/월 생성 버튼 등) 명확히 잘라두기.
