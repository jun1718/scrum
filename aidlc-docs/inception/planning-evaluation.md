## 2. AI-DLC 6대 영역 심층 평가

### 2.1 Functional Requirements — 6.5/10

**강점**
- 08-사용자-행동별-기능정의.md에 팀장/팀원/공통/배치 4개 actor 행동이 체계적으로 정의됨
- 07-보고-생성-프로세스.md의 데일리→주간→월간→성과 계층이 명확함
- 12-에러-유효성-메시지.md에 핵심 유효성 검증 6건 정의

**발견된 갭**

| # | 갭 | 근거 | 심각도 |
|---|-----|------|--------|
| F-1 | API 스펙이 "후보 목록" 수준 | 11-API-후보-목록.md에 "경로/메서드는 구현 시 확정" 명시. Request/Response JSON 없음 | 높음 |
| F-2 | "+" 수동 추가 업무의 taskId/taskLink 처리 미정의 | 08에서 "+" 직접추가 허용, DDL에서 report_detail.task_id NOT NULL — 두레이 외 업무의 ID 채번 방법 없음 | 높음 |
| F-3 | 팀 삭제, 멤버 탈퇴/팀 이동 시나리오 완전 부재 | 08에 생성만 존재, 삭제/이동 흐름 없음 | 중간 |
| F-4 | 성과 보고 기간 겹침 검증 로직 미정의 | 12에 에러 메시지만 존재, staDate/endDate overlap 판정 SQL/로직 없음 | 낮음 |
| F-5 | 동시 편집/경쟁 조건 처리 규칙 부재 | "주간보고 생성 전까지 수정 가능"은 있으나, 배치 실행 중 수정 시도 처리 미정의 | 낮음 |

- "트랜잭션 단위: 멤버 1명당 1건"으로 배치 격리 수준 정의됨

**발견된 갭**

| # | 갭 | 상태 |

| N-1 | 성능 목표 (API 응답시간, 동시 사용자 수) | 전혀 정의 안 됨 |
| N-2 | 가용성/SLA 목표 | 전혀 정의 안 됨 |
| N-3 | 배치 실패 시 처리 (재시도? 수동 트리거? 알림?) | 전혀 정의 안 됨 |

**강점**

## 3. 보안 기준선 평가 (SECURITY-01~15)

> 스프린톤 PoC 기준으로 평가. 사내 적용 시 필수 항목(*) 별도 표기.

| 규칙 | 항목 | 기획 반영 | 심각도 |
|------|------|-----------|--------|
| SECURITY-01 | 암호화 (전송/저장) | JWT claim에 dooray_api_token 암호화 언급. DB 암호화 미언급 | 중간 |
| SECURITY-02 | 네트워크 접근 로깅 | 미언급 | 낮음 |
| SECURITY-03 | 앱 레벨 로깅 | 로깅 전략 자체가 부재 | 중간 |
| SECURITY-04 | HTTP 보안 헤더 | CSP, HSTS, X-Frame-Options 등 미언급 | 중간 |
| SECURITY-05* | 입력 검증 | 12에 일부(taskLink 형식 등). 체계적 전략 부재 | **높음** |
| SECURITY-06* | 최소 권한 | 팀장/팀원 권한 분리 메커니즘 미정의 | **높음** |
| SECURITY-07 | 네트워크 구성 | 인프라 미정의로 해당 없음 | N/A |
| SECURITY-08* | 접근 제어 (IDOR) | JWT 검증만 언급. 타인 보고서 접근 방지(IDOR) 미고려 | **높음** |
| SECURITY-09 | 보안 하드닝 | 에러 응답 스택트레이스 노출 방지 미언급 | 중간 |
| SECURITY-10 | 소프트웨어 공급망 | 미언급 | 낮음 |
| SECURITY-11 | 보안 설계 원칙 | Rate limiting 미고려. 인증 로직 분리 미설계 | 중간 |
| SECURITY-12* | 인증/자격 관리 | JWT 기반 인증 정의됨. 세션 만료/로그아웃/브루트포스 보호 미정의 | **높음** |
| SECURITY-13 | 무결성 검증 | 미언급 | 낮음 |
| SECURITY-14 | 알림/모니터링 | 미언급 | 낮음 |
| SECURITY-15 | 예외 처리/fail-safe | 12에 기본 에러만. 글로벌 에러 핸들러 미설계 | 중간 |

**보안 종합: 4.0/10** — JWT 인증 기본 설계만 존재. 권한 분리, IDOR 방지, 체계적 입력 검증, 세션 관리가 모두 미정의.

## 4. 문서 간 일관성·DDL 정합성 교차 검증

### 4.1 DDL ↔ 기획 문서 정합성

| 검증 항목 | 결과 | 상세 |
|-----------|------|------|
| 06(테이블 정의) ↔ ddl.sql | **일치** | 7테이블, 컬럼명, 타입 일치 |
| 13(한글 매핑) ↔ ddl.sql | **일치** | 컬럼 매핑 정확 |

### 4.2 발견된 불일치

| # | 불일치 | 문서 A | 문서 B | 해결 방안 |
|---|--------|--------|--------|-----------|
| C-1 | report_detail.tag_id NOT NULL vs 태그 미선택 가능 | ddl.sql:71 `tag_id BIGINT NOT NULL` | 08 "행 입력" 흐름에서 태그 선택이 필수인지 불명확 | 태그 필수 여부 확정. 선택이면 DDL에서 NULL 허용 |
| C-2 | report_detail.task_id NOT NULL vs 수동 추가 업무 | ddl.sql:68 `task_id BIGINT NOT NULL` | 08 "+" 직접추가 기능 — 두레이 외 업무에 task_id 없음 | 수동 추가 시 task_id 채번 규칙 정의 또는 NULL 허용 |
| C-3 | 팀장/팀원 역할 구분 vs role 컬럼 부재 | 08 "팀장: 팀 생성, 태그 설정" | member 테이블에 role 없음 | role 컬럼 추가 또는 team.created_member_id=팀장 규칙 확정 |

### 4.3 기능 ↔ API 매핑 검증

08(기능정의)에 정의된 행동 vs 11(API 후보)에 매핑된 API:

| 기능 | API 매핑 | 상태 |

| 태그 설정 | POST /teams/{teamId}/tags | OK (수정/삭제 API 부재) |
| 팀 지정 | POST /members/me/team | OK |
| 데일리 스크럼 저장 | POST /reports/daily | OK |
| 주간/월간 보고 조회 | GET /reports/{type} | OK |
| **태그 수정/삭제** | **없음** | **갭** |
| **팀 삭제** | **없음** | **갭** |
| **멤버 탈퇴/이동** | **없음** | **갭** |

| 평가 깊이 | 영역당 1~2줄 | 영역당 강점+갭 테이블+구체적 근거 |
| 보안 평가 | 없음 | SECURITY-01~15 전 항목 개별 평가 |
| 교차 검증 | 없음 | DDL↔기획, 기능↔API 정합성 3건 불일치 발견 |
| 점수 경향 | 관대 (종합 7.5~8) | 구체적 갭 반영 (종합 6.3) |
| 권고사항 | "API 예시 확정", "성능 목표 보강" 등 추상적 | P0~P3 우선순위별 19건 구체적 조치 |
| Edge case | 미언급 | 7건 식별 (U-1~U-7) |

**가장 큰 점수 하락 요인**: Non-Functional Requirements (6~7 → 4.5). 성능, 가용성, 외부 API 장애 대응, 배치 실패 처리가 사실상 전무.

**가장 큰 리스크**: DDL-기획 불일치 3건(C-1~C-3). 구현 착수 후 발견 시 스키마 마이그레이션 필요.

---

## 8. 결론

기획 문서는 **비즈니스 컨텍스트와 데이터 모델에서 강점**을 보이나, **비기능 요구사항, 보안 설계, edge case 처리가 전반적으로 부족**하다. 특히 DDL-기획 간 불일치 3건(P0)은 구현 전 반드시 해결해야 한다.

스프린톤 2일 제약을 고려하면 P0 4건만 즉시 해결하고 P1~P3는 구현 중/후 순차 보강하는 것이 현실적이다.
